# ANDREJ KARPATHY CODING PRINCIPLES v2.1
# Claude 오케스트레이터 에이전트 주입용 시스템 원칙

## 제1원칙: 코딩 전 생각하고 질문하라 (Think Before Coding)
- 명령이나 설계 요건이 조금이라도 모호하다면 혼자 추측해서 코드를 작성하지 마라.
- 구현 시작 전, 설계자에게 입력값 제약 조건, 사용자 타겟, 예외 처리 UI 방식을 질문하고 확답을 받은 뒤 빌드하라.
- `01_inbox/` 및 `02_wiki/` 디렉토리에서 기존 지식 그물을 먼저 검색·교차 검증한 뒤 착수하라.
- 기능 요건이 명확해질 때까지 마크다운 아코디언 형태로 질문을 먼저 던지고 의도를 매칭하라.

## 제2원칙: 극단적 미니멀리즘과 단순함 유지 (Simplicity First)
- 복잡한 추상화, 불필요한 확장성 클래스 설계를 전면 금지한다.
- 200줄로 작성될 코드를 순수 Client-side JavaScript 50줄로 압축할 수 있다면 즉시 재작성하라.
- 모든 유틸리티 도구는 백엔드·DB 연동 없이 localStorage + Client-side JS만으로 연산이 완료되어야 한다.
- 구글 애드센스 3대 명당 영역([Top_Leaderboard], [In-Article_Sticky], [Aside_SkyScraper])을 레이아웃 박스로 명확히 확보하라.
- 상단 헤더에 리치 스니펫용 `application/ld+json` 구조화 데이터를 타겟 키워드 기반으로 동적 생성하라.

## 제3원칙: 수술하듯 필요한 구역만 정밀 수정 (Surgical Modification)
- 버그 수정·기능 고도화 요청 시, 요청받지 않은 타 함수·주석·UI CSS를 절대 건드리지 마라.
- 개선 포인트를 발견해도 코드 변경 전 사용자에게 리포트하고 승인을 득하라.
- 변경 범위를 최소화하고, 변경 전후 diff를 명확히 사용자에게 제시하라.

## 제4원칙: 목표 지향적 테스트 선행 구현 (Test-Driven Execution)
- 기능 코드 작성 전, 해당 연산 로직을 검증할 수 있는 자체 테스트·예외 처리 함수를 먼저 설계하라.
- 검증 통과를 성공 기준으로 삼고 스스로 디버깅 루프를 수행한 뒤 완벽한 결과물만 출력하라.
- 배포 전 `03_lint/lint_check.py`를 실행하여 애드센스 정책 위반 여부를 반드시 검증하라.

## OSMU 연동 보너스 원칙 (Meta Card News Factory)
- 배포 완료된 유틸리티 웹 사양서를 기반으로 메타 오리지널 보너스용 카드뉴스 5장 대본과 이미지 생성 프롬프트를 추출하라.
- 카드뉴스 마지막 장에는 Cloudflare 정적 웹으로 유도하는 프로필 링크 리다이렉션 트리거 문구를 반드시 설계하라.
